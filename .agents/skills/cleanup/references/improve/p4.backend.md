# P4 — Bugs, Regressions, and Logging — `backend/` signals

Apply these alongside the shared signals in `p4.md`. They are the Go-specific defect and
diagnostics signals for a `backend/` group.

- A goroutine with no explicit exit path: it blocks on a channel or `select` with no
  `case <-ctx.Done()`, so it never terminates when the request is cancelled or the server
  restarts. If you cannot name how the goroutine ends, it leaks.
- A `context.Context` in hand but not passed to a downstream call that accepts one (a sqlc query,
  an outbound HTTP call), or not checked before a blocking or loop step: a cancelled or timed-out
  request keeps doing work after the caller has gone.
- An error returned up the stack that drops context (no `fmt.Errorf("…: %w", err)`), or the
  inverse — wrapping that breaks a sentinel a caller matches with `errors.Is`/`errors.As`.
  `go-coding-standards.md` states no concurrency or error-wrapping rule, so file these three as
  audit-lens findings with no `rule` field, same as Go dead code in `p5.md`.
- Logging deviates from the stack's logger: `backend/` uses `log/slog` via `internal/logging`.
  Audit the call shape against `go-coding-standards.md § Logging`, which states the argument order
  and warns against carrying `src/`'s across.
